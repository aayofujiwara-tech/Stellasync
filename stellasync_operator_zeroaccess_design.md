Stellasync（ナイト系向けX SNS解析SaaS）について、預かった他人のXアクセストークンを、
開発者かつ運用者である私自身からも「可能な限り見えない」状態に作り替えたい。
これはセキュリティ監査とは別タスク（既存コードの穴探しではなく、鍵管理・権限・
監査ログのアーキテクチャ設計）として進める。

【前提・コンテキスト】
- スタック: React+Vite / Firebase(Auth匿名+X OAuth, Firestore asia-northeast2) /
  Cloud Functions v2(TypeScript) / Cloudflare Pages / Google Secret Manager
- 現状: トークンは自前 ENCRYPTION_KEY を使った AES-256-GCM(crypto.ts)で暗号化し
  Firestore に保管。鍵は Secret Manager
- サービスは定期ポーリング(15分/2時間/日次)でキャストのトークンを使い X API を叩く

【最重要・脅威モデルの前提（必ず守る）】
- このサービスは「サーバー(Functions)が定期的にトークンを復号して使う」必要がある。
  かつ GCP プロジェクトとサーバーを握るのは運用者本人。
  したがって「運用者が技術的に絶対アクセスできない（完全ゼロ知識）」は原理的に不可能。
  これを"可能"であるかのように設計・説明しないこと。
- 本タスクの目標は、完全な不可能化ではなく、以下4点の達成とする：
  (1) 平文の鍵がどこにも存在しない
  (2) コンソール等から「素で」トークンを読める経路が無い（読むにはコード実行が必要）
  (3) 復号アクセスは必ず改ざん耐性のある監査ログに残る
  (4) 日常の管理ツール・ログに生トークンが一切出ない
- 脅威モデルは「運用者＝潜在的攻撃者／内部不正・端末侵害」。運用者を信頼する前提の
  設計になっていないかを常にチェックする。

【絶対ルール】
- フェーズA（調査・設計提案）は読み取り専用。コード変更・デプロイ・IAM変更・
  データ操作は一切しない
- 設計提案は「現状の事実」「変更案」「実現できること／原理的にできないこと」を
  明確に区別して書く。推測は推測と明記
- 実装はフェーズBで、私が承認した項目のみ、最小変更で実施する

【A-1. 現状の鍵管理・復号経路の把握】
- crypto.ts と Functions 内で、どこで・どの権限で ENCRYPTION_KEY を取得し復号しているか
- 現状、運用者の個人アカウント or サービスアカウントのどちらの権限で鍵にアクセスできるか
  （※IAMの実体はコンソール側。コードから読める範囲＝参照箇所のみ把握し、
    IAMの実権限は「要手動確認」として切り分ける）
- 生トークン or 復号後トークンが、ログ・レスポンス・管理用エンドポイントに
  出ていく経路が無いか（A-5の前提として洗い出す）

【A-2. Cloud KMS エンベロープ暗号化への移行案】
- 自前 ENCRYPTION_KEY を廃し、Cloud KMS の鍵で DEK(データ暗号鍵)を保護する
  エンベロープ暗号化へ移行する設計を提案
- 平文の鍵がディスク/Secret/環境変数に存在しなくなることを設計上担保
- crypto.ts の変更点、暗号文フォーマットの変更点、既存の暗号化済みトークンを
  新方式へ移し替える再暗号化マイグレーションの手順案（ダウンタイム/ロールバック含む）

【A-3. IAM分離案（人間の自分 ≠ サービスの自分）】
- 復号できるのは Functions のサービスアカウントのみ。私の個人Googleアカウントには
  KMS の decrypter / Secret の secretAccessor を付けない構成を提案
- これにより「コンソールのボタンで素でトークンを覗く」経路が消えることを説明
- ※IAM の付与・剥奪は私が手動で実施する。Claude Code は変更しない。
  「私が手動で実行すべき gcloud コマンド / コンソール操作」を手順として出力する

【A-4. 改ざん耐性のある監査ログ】
- KMS の decrypt 操作に対する Cloud Audit Logs(Data Access)を有効化する案
- ログを本番プロジェクトとは別プロジェクト or 組織レベルのログシンクへ転送し、
  保持期間ロックをかけ、運用者が後から黙って削除できないようにする案
- 復号アクセス発生時に私へ通知(アラート)が飛ぶ仕組みの案
- これらは大半がコンソール/組織設定側。コードで対応できる部分（例: 構造化ログの
  出力方針）と、私が手動設定する部分を分けて提示

【A-5. データ最小化・生トークンの遮断】
- 復号は Functions 内に閉じ、復号後の平文トークンをクライアントや管理画面・
  レスポンスへ一切返さない設計になっているか / するための変更案
- 管理用ダッシュボードは集計(daily_metrics 等)のみを扱い、生トークンを参照する
  adminエンドポイントを作らない方針の明文化
- 全ログでトークン/鍵をマスクする共通ユーティリティの導入案
- エラー時にトークンがスタックトレース等で漏れない設計か

【A-6. 本番分離・break-glass（ソロ運用の現実解）】
- 本番GCP/Firebaseプロジェクトを開発用と分離し、本番IAMを最小化する案
- 二者承認はソロでは不可能。代替として「復号権限の常時付与をやめ、必要時のみ
  一時付与(short-lived) → 操作 → 即時剥奪」かつ「付与/操作が必ず自分に
  アラートされる」break-glass運用の手順案
- この運用が「絶対防げる」ではなく「素では起きない＋必ず痕跡が残る」止まりである
  ことを正直に明記
