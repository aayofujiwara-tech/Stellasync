Stellasync（ナイト系向けX SNS解析SaaS）を知り合いの店舗に有償提供する前に、
セキュリティレビューを実施したい。他人のXアカウントのアクセストークン（暗号化保管）を預かり、
複数店舗のデータを扱うサービスなので、課金開始前の必須条件として徹底的に点検してほしい。

【前提・コンテキスト】
- スタック: React+Vite / Firebase(Auth匿名+X OAuth, Firestore asia-northeast2) /
  Cloud Functions v2(TypeScript) / Cloudflare Pages / Google Secret Manager
- 認証: 匿名Auth + X OAuth 2.0 PKCE。トークンはAES-256-GCMで暗号化しFirestoreに保管
- マルチテナント想定: accounts に org_id / store_id / cast_id フィールドあり
- 直近でテスト用 debugTrigger を作成・削除した経緯あり（残骸が無いか要確認）
- 過去にサービスアカウント鍵(stellasync-app-firebase-adminsdk-*.json)をルートに置いて
  いた時期があり、後で削除した（git履歴に残っていないか要確認）

【絶対ルール】
- フェーズ1（調査）は読み取り専用。コード変更・デプロイ・データ操作は一切しない
- 各指摘は「ファイル:行」を引用し、深刻度(Critical/High/Medium/Low)を付ける
- 修正はフェーズ2で、私が深刻度と内容を確認し承認したものだけ実施する
- 推測と確認事実を区別する。コードから断定できないものは「推測」「要手動確認」と明記し、
  事実であるかのように書かない
- リポジトリから確認できない項目（GCP/Firebaseコンソール側のIAM・課金設定・
  App Check有効化状態など）は、フェーズ1の最終報告で「確認不能・要手動チェック項目」
  として別枠にまとめる。コードだけ見て「問題なし」と誤って結論づけない

【1. Firestoreセキュリティルール（最重要）】
firestore.rules を精読し、以下を検証：
- accounts: 各ユーザーが自分のドキュメントのみ read/write でき、他人の
  access_token/refresh_token を読めないか。トークンフィールドがクライアントから
  読み取り可能になっていないか（暗号化済みでも漏らすべきでない）
- post_hourly_metrics / daily_metrics: cast_id == request.auth.uid 等で
  他人のデータを読めない設計か
- マルチテナント分離: A店のキャストがB店(別org_id/store_id)のデータに
  アクセスできない保証があるか。org_id/store_id ベースの分離ルールが
  実際に書かれているか、それとも未実装で「今は1店舗だから事故ってないだけ」か
- oauth_sessions: state/code_verifier を含むこのコレクションが
  クライアントから読めてしまわないか（読めると CSRF/セッション奪取リスク）
- デフォルト deny になっているか。ルールに穴(allow read: if true 等)がないか
- 匿名Auth前提の権限設計: 匿名uidは無制限に生成できる。ルールが匿名ユーザーに
  対して余計な read/write 権限を与えていないか（認証済み=信頼できる、という
  前提になっていないか）
- ルールがエミュレータでテストされているか。@firebase/rules-unit-testing 等で
  「A店キャストがB店データを読めない」「他人のaccountsを読めない」を実際に
  検証するテストコードが存在するか。無ければ「目視レビューのみ＝ロジックバグ
  見落としリスクあり」と明記

【2. トークン・シークレットの取り扱い】
- crypto.ts: AES-256-GCM の実装が正しいか（IV/nonceの生成が毎回ランダムか、
  認証タグの検証をしているか、鍵長が適切か、暗号文と一緒にIV/タグを保存しているか）
- ENCRYPTION_KEY/X_CLIENT_SECRET 等が Secret Manager 経由のみで参照され、
  コード/ログ/レスポンスに平文で出ていないか
- アクセストークン/リフレッシュトークンが console.log やエラーレスポンスに
  漏れていないか（functions/src 全体を grep）
- 復号済みトークンがクライアントに返っていないか: いずれかのエンドポイントが
  復号後の平文アクセストークンをレスポンスに含めていないか。復号はFunctions内に
  閉じ、平文トークンはクライアントへ一切返さない設計になっているか（独立項目として確認）
- フロントエンド(src/)に秘密値がハードコードされていないか
  （VITE_ 環境変数で公開されるべきでないものが混ざっていないか）

【3. debugTrigger 残骸・危険な公開エンドポイント】
- functions/src に debugTrigger.ts が残っていないか、index.ts に export が
  残っていないか
- onRequest で公開されている全関数(authXRedirect/authXCallback等)を列挙し、
  認証・CORS・入力検証が適切か。特に authXCallback が state検証・
  セッション有効期限・redirect_uri検証を正しく行っているか
- uid をクエリで受け取る関数が、なりすまし(他人のuidを指定)を許さないか
- CORS が `*`(全許可)でなく、Cloudflare Pages のオリジンを allowlist で
  限定しているか
- 注: Functions(v2/Cloud Run)の「未認証呼び出し許可(invoker IAM)」は
  コードからは確認できない。重い処理を無認証で叩かれる課金・乱用リスクがあるため、
  これは【7】の手動チェック項目として報告に回すこと

【4. OAuthフローの安全性】
- PKCE の code_verifier/code_challenge(S256)が正しく実装されているか
- state パラメータで CSRF を防いでいるか、セッションが使い捨て(一度使ったら削除)か
- oauth_sessions の TTL(10分)が機能し、期限切れセッションが拒否されるか
- トークン失効監視(tokenWatcher)とリフレッシュ(refreshXToken)に
  ロジック上の穴がないか
- X OAuth の要求スコープが最小化されているか。解析用途に不要な書き込み系スコープ等を
  要求していないか（過剰スコープ＝トークン漏洩時の被害が拡大する）

【5. git履歴・機密ファイル混入】
- .gitignore に .env.local / サービスアカウント鍵(*-adminsdk-*.json) /
  .env が含まれているか
- git log/git のトラッキング状態で、過去にコミットされた秘密情報が
  履歴に残っていないか確認する方法を提示
  （git log --all --full-history -- "*adminsdk*.json" 等で痕跡確認）
- 現在の作業ツリーやステージに秘密ファイルが含まれていないか
- 【重要】サービスアカウント鍵やトークンが過去コミットに1度でも含まれていた場合、
  ファイルを削除しても鍵自体は有効なまま＝漏洩扱いになる。その場合の対応として
  「GCPコンソールで該当鍵を無効化・ローテーション」「ENCRYPTION_KEY等の
  ローテーション」「履歴の書き換え(git filter-repo / BFG)」が必要であることを
  報告に明記する。"履歴から消せばOK"で終わらせない

【6. 依存パッケージ・その他】
- npm audit 相当で既知の脆弱性がある依存がないか（functions と root 両方）
- Firestore のインデックス・ルールがデプロイ済みの状態と
  ローカル定義で乖離していないか

【7. コンソール側・要手動チェック項目（リポジトリから確認不能）】
コードからは判定できないが課金前に必須の項目。Claude Code は確認できないため、
「私(運用者)が手動で確認すべきチェックリスト」として、確認手順つきで出力すること：
- Cloud Functions(v2/Cloud Run)の各エンドポイントの invoker IAM。
  未認証呼び出しを許可していないか（公開して良いのは authX系など最小限か）
- Secret Manager のアクセスIAM が最小権限か（不要なプリンシパルに
  secretAccessor を付けていないか）
- 請求アラート/予算上限が設定されているか（Firestore読み取り・Functions呼び出しの
  乱用によるコスト爆撃対策）
- Firebase App Check が有効か。アプリ以外のクライアントから Firestore/Functions を
  直接叩く乱用を防げているか。有償マルチテナントでは実質必須のため、
  未導入なら High 相当として扱う
